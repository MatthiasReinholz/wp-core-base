# wp-core-base 1.4.2

## Summary

This is the `patch` framework release from `v1.4.1` to `v1.4.2` for `wp-core-base`.

It hardens the framework self-update flow by turning `framework-sync` into a safer preflightable operation.

The main change is a reusable framework install plan that powers richer `framework-sync --check-only` reporting, optional machine-readable JSON output, and a new strict mode that can fail early when customized framework-managed files would be skipped.

This release also fixes the strict refresh path so the new preflight and strict-mode behavior stays accurate when framework-managed files need to refresh during a real update.

## Downstream Impact

Downstream repositories pinned to an older `wp-core-base` release can update to `v1.4.2` through `framework-sync` or by vendoring the published snapshot manually.

There is no breaking migration for existing GitHub or GitLab downstreams:

- default `framework-sync` behavior remains unchanged unless you opt into the new strict flag
- existing framework-managed workflow skip preservation still works
- the new output is additive and meant to make framework updates safer to review and automate

Operationally, downstreams now get a clearer preflight surface before applying a framework update:

- `framework-sync --check-only` now reports release scope, release metadata, and planned managed-file refresh/remove/skip actions
- `framework-sync --check-only --json` can be used in CI or GitOps-style preflight checks
- `framework-sync --fail-on-skipped-managed-files` can fail early when local workflow customizations would otherwise be preserved silently

## Migration Notes

No special migration steps are required for existing downstreams that continue to use the default framework update flow.

If your downstream repository has locally customized framework-managed workflow files, this release gives you a safer way to review those updates before rollout:

1. run `framework-sync --check-only` locally first
2. if you automate framework adoption, prefer `framework-sync --check-only --json`
3. if you want framework updates to stop instead of preserving customized managed files, add `--fail-on-skipped-managed-files`

## Downstream Workflow Changes

`v1.4.2` does not change the scaffolded GitHub workflows or GitLab pipeline templates themselves.

The downstream-facing workflow change is in how framework updates can now be preflighted:

- framework update checks now expose planned refreshed, removed, and skipped managed files before apply
- strict mode can now fail a framework update when customized framework-managed files would be skipped
- the planner uses the downstream automation provider when classifying managed files, so GitHub and GitLab downstreams get accurate framework-update previews

## Required Downstream Actions

No required downstream action exists for existing GitHub or GitLab downstreams.

Recommended follow-up for teams that automate framework upgrades:

1. add a preflight step based on `framework-sync --check-only`
2. use `--json` if the result is consumed by CI or another automation layer
3. use `--fail-on-skipped-managed-files` if customized framework-managed files should block rollout instead of being preserved
