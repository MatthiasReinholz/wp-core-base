# wp-core-base 1.4.0

## Summary

This is the `minor` framework release from `v1.3.4` to `v1.4.0` for `wp-core-base`.

This release makes multi-host automation a first-class part of the framework contract for `GitHub` and `GitLab` while keeping the staged WordPress runtime payload provider-neutral.

The main architectural change is an explicit split between:

- the automation host where update PRs or MRs run
- the managed dependency release source
- the authoritative framework release source recorded in `.wp-core-base/framework.php`

That separation makes it possible to run automation on one supported host, consume managed dependency archives from another supported release source, and keep the framework's own official publication source singular and explicit.

## Downstream Impact

Existing GitHub downstreams remain the compatibility baseline:

- GitHub stays the default automation provider
- existing `github-release` dependency definitions continue to work
- legacy framework metadata that still carries only `repository` continues to load
- framework-managed workflow refreshes remain reviewable through normal PR flow

GitLab is now a first-class automation host for downstreams:

- `scaffold-downstream` can now generate `.gitlab-ci.yml`
- sync, blocker, reconciliation, sync-report, and framework self-update flows can run against GitLab
- self-managed GitLab is supported through explicit API base configuration

Managed dependency release sourcing is now broader and cleaner:

- `wordpress.org`, `github-release`, `gitlab-release`, and downstream `premium` providers are supported
- automation host selection is no longer coupled to hosted release source selection

Runtime and deployment behavior stay intentionally host-neutral:

- provider-specific CI metadata stays in the repository scaffolding layer
- staged runtime payloads exclude provider-specific CI files such as `.github`, `.gitlab-ci.yml`, `.gitea`, `.forgejo`, and `bitbucket-pipelines.yml`

## Supported Git Environments

First-class hosted automation support in this release:

- `GitHub`
- `GitLab`

Supported with explicit configuration:

- `GitHub Enterprise`
- self-managed `GitLab`

Supported only for the repository/runtime model, not first-class hosted automation:

- other Git hosts where teams still want the manifest model, deterministic staging, and manual framework adoption

Not yet implemented as native automation providers:

- `Gitea`
- `Forgejo`
- `Bitbucket`

## Key Changes

- Added provider-aware automation abstractions so update orchestration no longer assumes GitHub internally.
- Added GitLab automation support for PR-equivalent flows, blocker evaluation, reconciliation, sync reporting, and framework self-update.
- Added `gitlab-release` as a managed dependency source alongside `github-release`.
- Made the framework's authoritative release source explicit in `.wp-core-base/framework.php`.
- Kept the framework release source singular. The current official framework release source remains `GitHub Releases`.
- Added GitLab-specific downstream scaffolding and examples.
- Tightened runtime hygiene so provider-specific CI metadata does not leak into staged WordPress payloads.
- Hardened security and operator ergonomics discovered during review:
  - redacted host-client response bodies and credential-bearing URLs in user-facing output
  - removed stale framework-managed files when provider-managed file sets change
  - preserved locally customized legacy workflow files during framework self-update
  - rejected unknown CLI flags early instead of failing later with unrelated errors
  - isolated doctor staging paths so concurrent doctor runs do not collide

## Migration Notes

No special migration is required for downstreams already using `GitHub`.

Recommended adoption guidance:

- GitHub downstreams can update directly and continue using the existing automation contract.
- GitLab downstreams should review the scaffolded `.gitlab-ci.yml`, confirm `GITLAB_TOKEN` and project metadata are available in CI, and treat the generated pipeline as framework-managed unless intentionally customized.
- Downstreams with locally customized framework-managed files should continue reviewing workflow diffs during `framework-sync` instead of assuming silent overwrite.

Important scope boundaries in `v1.4.0`:

- the framework's official release source is still singular and still published from `GitHub Releases`
- this release does not add native automation providers for `Gitea`, `Forgejo`, or `Bitbucket`
- this release does not change the update model to WordPress in-dashboard updates; managed updates remain Git-reviewed PR or MR flows
