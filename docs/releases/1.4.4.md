# wp-core-base 1.4.4

## Summary

This is the `patch` framework release from `v1.4.3` to `v1.4.4` for `wp-core-base`.

It makes Akismet a workflow-managed WordPress.org dependency, updates the bundled Akismet baseline to `5.7`, hardens GitHub automation PR creation, adds release-key lifecycle policy checks, and tightens the upstream repository security posture.

## Downstream Impact

Downstream repositories pinned to an older `wp-core-base` release can update to `v1.4.4` through `framework-sync` or by vendoring the published snapshot manually.

Review framework-managed workflow refreshes carefully: GitHub update workflows now prefer `WP_CORE_BASE_AUTOMATION_TOKEN` so generated PRs can trigger required pull-request checks.

## Migration Notes

GitHub-hosted downstreams should add a repository secret named `WP_CORE_BASE_AUTOMATION_TOKEN` with permission to push update branches and open pull requests.

If that secret is absent, scaffolded workflows still fall back to `GITHUB_TOKEN`, but generated PRs may not trigger required checks. Downstreams with locally customized scaffolded workflow files should review upstream workflow diffs manually before adopting this release.

## Downstream Workflow Changes

The GitHub update, update-reconciliation, and framework self-update templates now pass `WP_CORE_BASE_AUTOMATION_TOKEN` to checkout and the updater CLI when configured. They also emit an explicit warning when falling back to `GITHUB_TOKEN`.

## Required Downstream Actions

Configure `WP_CORE_BASE_AUTOMATION_TOKEN` before relying on scheduled GitHub update PRs with required checks. If your downstream has customized framework-managed files, run `framework-sync --check-only --fail-on-skipped-managed-files` before rollout and review skipped workflow files manually.

## Bundled Baseline

- WordPress core: `6.9.4`
- WooCommerce `10.7.0`
- Jetpack `15.7.1`
- Contact Form 7 `6.1.5`
- Redirection `5.7.5`
- Akismet Anti-spam: Spam Protection `5.7`

## Operational Notes

The published framework asset for this release is `wp-core-base-vendor-snapshot.zip`.

Normal release flow: run `prepare-wp-core-base-release`, review and merge `release/v1.4.4`, then let `finalize-wp-core-base-release` create the tag and publish the authoritative framework release artifact.

Release signature verification now reads committed public-key metadata and revocation data. Revoked keys fail verification; retired or expired keys produce warnings so maintainers can rotate without breaking historical verification immediately.
