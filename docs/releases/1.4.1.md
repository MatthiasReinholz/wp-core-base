# wp-core-base 1.4.1

## Summary

This is the `patch` framework release from `v1.4.0` to `v1.4.1` for `wp-core-base`.

This release adds a new managed dependency source for host-agnostic plugin and theme distribution while preserving the existing `GitHub` and `GitLab` release-backed automation contract introduced in `v1.4.0`.

The main change is built-in support for `source: 'generic-json'`, which lets downstreams consume installable plugin or theme archives from a stable public HTTPS metadata endpoint when the upstream project is not publishing its artifacts as GitHub or GitLab Releases.

This release also closes one reliability issue found during the final release review: runtime cleanup is now more defensive when removing temporary build and verification trees, which avoids intermittent non-empty-directory failures during repeated verification runs.

## Downstream Impact

Downstream repositories pinned to an older `wp-core-base` release can update to `v1.4.1` through `framework-sync` or by vendoring the published snapshot manually.

There is no breaking migration for existing GitHub downstreams:

- GitHub remains the default automation provider
- existing `github-release` dependencies continue to work unchanged
- the new `generic-json` behavior is only active for dependencies that explicitly opt into `source: 'generic-json'`

New source surface in `v1.4.1`:

- `generic-json` is now supported for managed `plugin` and `theme` dependencies
- the metadata endpoint must be public HTTPS
- the metadata endpoint must advertise a current installable version plus a real release timestamp
- the source is latest-version only; it is not a historical release catalog

Operational impact:

- authoring now supports `--source=generic-json` and `--generic-json-url=...`
- updater PRs from this source are labeled and rendered as `generic-json` updates instead of falling back to WordPress.org wording
- superseded open PRs for older `generic-json` versions are closed automatically once the metadata feed advertises a newer release

## Migration Notes

No special migration steps are required for existing GitHub or GitLab downstreams.

If you want to adopt the new metadata-driven source:

1. confirm the upstream endpoint publishes a stable HTTPS metadata document
2. confirm it exposes `version`, `download_url`, and one valid release timestamp field such as `release_at`, `published_at`, `last_updated`, or `updated`
3. switch the dependency definition to `source: 'generic-json'` with `source_config.generic_json_url`
4. leave verification at `inherit` or `none`; checksum-sidecar hardening is not supported for `generic-json` today

If your downstream repository has locally customized scaffolded workflow files, continue reviewing framework-managed workflow diffs manually before adopting this release.

## Downstream Workflow Changes

`v1.4.1` does not change the scaffolded GitHub or GitLab automation templates themselves.

The main operational change is around dependency-source handling and release verification:

- downstreams may now declare managed `plugin` and `theme` dependencies with `source: 'generic-json'`
- updater PRs from that source now carry `generic-json`-specific labels and wording
- runtime cleanup during framework verification is more defensive, which reduces intermittent maintainer-side release verification noise but does not change downstream workflow files

## Required Downstream Actions

No required downstream action exists for existing GitHub or GitLab downstreams that do not adopt `generic-json`.

If you do adopt `generic-json`:

1. confirm the metadata endpoint is public HTTPS
2. confirm it publishes a current installable ZIP plus a real release timestamp field
3. update the dependency definition to include `source_config.generic_json_url`
4. keep verification at `inherit` or `none`, because checksum-sidecar hardening is not supported for this source today

If you carry local modifications in framework-managed workflow files, continue checking framework updates locally before merge and reconcile those files manually when a future release changes the upstream templates.

## Key Changes

- Added `GenericJsonManagedSource` as a built-in managed source for public HTTPS plugin and theme metadata endpoints.
- Added config validation and normalization for `source: 'generic-json'` and `source_config.generic_json_url`.
- Added `add-dependency` and `adopt-dependency` support for `--source=generic-json` and `--generic-json-url=...`.
- Required real release timestamps for generic metadata sources so release-age and review metadata stay coherent.
- Treated `generic-json` as a latest-only source by pruning superseded open PRs instead of trying to refresh versions that the feed no longer advertises.
- Added source-specific updater labels and PR body wording for `generic-json`.
- Added downstream docs and examples for `wp-plugin-base` interoperability across GitHub Releases, GitLab Releases, and JSON metadata endpoints.
- Hardened `RuntimeInspector` cleanup with child-first directory removal, retry-based empty-directory handling, and exclusion-safe partial cleanup behavior.
